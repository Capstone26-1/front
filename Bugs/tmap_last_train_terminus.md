# [버그] Tmap 막차 종착역 미반영 — 실제 도달 불가 경로 반환

> 상태: **해결 완료** · 2026-06-04 `validate_transit_route` (Anthropic haiku 서브호출) 방식으로 수정
> 1차 수정(실시간 API 경유지 검증)은 실패. 최종 해결: `Taxi` 브랜치, 커밋 `dba6530`, `4184656`, `441e878`. 상세 구현 → `Features/validate_transit_route.md`

## 재현 쿼리

"인덕원역에서 돌곶이역 갈건데 23시 40분에 출발할거야"

## 증상

- 추천 경로: 인덕원 → (4호선) → **삼각지** → (6호선) → 돌곶이 (환승 포함)
- 실제 상황: 23:40 출발 기준 인덕원역 4호선 막차는 **사당 종착**
  → 사당 이북(동작·이촌·삼각지…)으로 가는 연결 열차가 없어 삼각지 도달 불가
- 그런데 위험도는 "안전/주의"로 낮게 표시되어 사용자를 오도함

## 실측 재현 결과 (2026-06-03, 실제 API 호출로 확인)

### 1. Tmap은 23:40에 깨진 경로를 실제로 반환한다

`POST https://apis.openapi.sk.com/transit/routes` (time=202606032340) 호출 결과 5개 경로 반환:

| 경로 | 구간 | 비고 |
|---|---|---|
| route 1 | 인덕원 →(4호선)→ **삼각지** →(6호선)→ 돌곶이 | 사당 이북 → 깨짐 |
| route 2 | 인덕원 →(4호선)→ **성신여대입구** →(우이신설)→ 보문 →(6호선)→ 돌곶이 | 사당 이북 → 깨짐 |
| route 3 | 인덕원 →(4호선)→ **이촌** →(경의중앙)→ 청량리 →(버스 N13)→ 석관시장 | 사당 이북 → 깨짐 |

→ Tmap은 정기 스케줄(이론상 연결) 기반이라 **막차가 중간 종착하는 사실을 반영하지 못함**.

### 2. Tmap 응답의 leg `departureTime`이 전부 `null`

각 구간(leg)의 출발/도착 시각 필드가 비어 있음 → **Tmap leg 시각으로 막차 여부를 검증하는 접근은 데이터가 없어 불가능**.

### 3. 실시간 지하철 API(`realtimeStationArrival`)는 "지금 이 순간" 열차만 반환

낮 시간에 인덕원역 조회 결과:

```
1004 | 상행 | 사당행 - 정부과천청사방면 | 종착 사당 | lstcarAt=0
1004 | 하행 | 안산행 - 평촌방면     | 종착 안산 | lstcarAt=0
```

- 모든 열차 `lstcarAt=0`(막차 아님) → **낮에 "23:40 출발" 시나리오를 테스트하면 막차 정보 자체가 없음**
- 상행(사당행)·하행(안산행)이 **섞여서** 반환됨 → 방향 구분 없이 첫 막차를 집으면 엉뚱한 방향을 잡을 수 있음

### 4. 서울 시간표 API는 역코드 체계가 달라 바로 못 씀

- 실시간 API의 `statnId`(예: 인덕원 = `1004000440`)를 시간표 서비스(`SearchSTNTimeTableByIDService`)에 넣으면 **빈 응답**
- 시간표 API를 쓰려면 **역명 → FR_CODE 매핑 테이블(~700개 역)**을 별도 구축해야 함

### 5. Tmap 일일 한도(QUOTA_EXCEEDED, 429) 존재

- 짧은 시간에 두 번 호출하니 한도 초과 발생 → **경유지마다 반복 검증 호출은 한도를 빠르게 소모**. 캐싱/호출 최소화 필요.

## 왜 1차 수정이 실패했는가

1차 수정은 "경유지(환승역)마다 `transit_disruption_tool`(실시간 API)을 호출해 막차 여부 확인 + Claude가 노선지식으로 도달 가능성 추론"하는 방식이었다. 하지만:

- 실시간 API는 **지금 운행 중인 열차**만 보여주므로, 낮에 "23:40 출발"을 테스트하면 막차 데이터가 없어 검증 로직이 **아예 작동하지 않음** (원인 #3)
- 환승역에서 `trains: []`(운행 종료)를 신호로 삼았으나, 환승역(삼각지)은 자체 노선(6호선) 열차가 돌아 `trains`가 비지 않음 → **잘못된 신호** (실시간 도착정보는 "그 역에서 곧 오는 열차"일 뿐, "인덕원발 열차가 거기까지 오는가"를 답하지 못함)

### 옵션 B — 서울 시간표 API 통합

- **장점**: 가장 정확, 전 노선 자동 커버, 수동 데이터 관리 불필요
- **단점**: 매핑 테이블 구축 + 현재 빈 응답 디버깅 필요(가장 무거움), 추가 API 호출

### 옵션 C — Claude 지식 단독 강화

- 새 데이터/API 없이, Tmap 경로 + 시각을 주고 Claude가 자체 서울 지하철 막차 지식만으로 "이 환승역까지 막차가 가는가"를 엄격히 추론하도록 프롬프트 강화
- **장점**: 가장 가벼움(코드/데이터 추가 없음)
- **단점**: 비결정적, 정확도가 Claude 지식에 전적으로 의존 (오답 위험)

## Q&A (팀 논의 중 정리)

### Q. Tmap 응답에서 "막차가 사당행"이라는 정보(열차 종착역)는 못 받아오나?

**A. 못 받아온다.**

1. **Tmap 지하철 leg가 주는 필드는 "내 승하차역"뿐**:
   - `route`: `"수도권4호선"` (노선명일 뿐, 행선지 아님)
   - `start.name` = 인덕원 (내가 **타는** 역), `end.name` = 삼각지 (내가 **내리는** 역)
   - `passStopList`: 인덕원~삼각지 사이 정차역 (내 구간만, 열차의 실제 종착역 아님)
   - `departureTime`: `null`
   → start/end는 "내 승하차역"이지 "열차의 종착역"이 아님. 열차가 사당에서 끊기는지 안 알려줌.

2. **논리적으로도 없을 수밖에 없음**: Tmap이 인덕원→삼각지를 4호선으로 이어붙인 경로를 반환했다는 건, 엔진이 "이 시각에 그 구간을 가는 열차가 있다"고 **믿는다**는 뜻. 막차 끊김을 알았다면 애초에 그 경로를 안 만들었을 것. **막차 끊김을 모르는 것이 버그의 본질** → Tmap 응답 안에 그 정보는 구조적으로 부재.

3. **"사당행" 정보의 실제 출처는 서울 실시간 API**(`trainLineNm`/`bstatnNm`). 단 이건 "지금 이 순간" 열차만 줘서, 낮에 "23:40 출발"을 테스트하면 23:40 막차의 종착역은 알 수 없음(원인 #3).

**→ 한 줄 결론**: Tmap도(종착역 필드 없음), 실시간 API도(미래 시각 조회 불가) "23:40의 막차 종착역"을 주지 못한다. 그래서 별도 막차 시간표 데이터(옵션 A/B)가 필요하다.

## Anthropic API 활용 해결 방안 (채택)

외부 API(실시간/시간표)의 한계를 Claude의 서울 지하철 지식으로 보완.

### 옵션 1: 시스템 프롬프트 강화 (Claude 지식 직접 활용)

- 추가 API 호출 없이, Tmap 경로 + 출발 시각을 주고 Claude 자체 서울 지하철 막차 지식으로 "이 환승역까지 막차가 가는가"를 엄격히 추론하도록 프롬프트 강화
- **장점**: 구현 부담 없음
- **단점**: 비결정적, 정확도가 Claude 지식에 의존

### 옵션 2: 경로 검증 전용 Claude 서브호출 (채택)

- 새 MCP 도구 `validate_transit_route` 추가
- 지하철 leg마다 Anthropic API(`claude-haiku-4-5-20251001`)를 서브호출해 막차 종착역 검증
- 도달 불가 판정 시: 종착역 좌표(Kakao API)를 구해 Tmap으로 재탐색
- 결과: `{hasInfeasibleLegs, lastReachableStation, altRoute}`
- 구현: `server/tools/subwayValidator.js` (신규) + `server/mcpServer.js` + `server/claude.js` 시스템 프롬프트
- **장점**: 낮/밤 언제 테스트해도 작동, 구조화된 JSON 출력으로 안정적
- **단점**: 지하철 leg당 haiku 호출 1회 추가

### 옵션 3: 실시간 API + Claude 해석 개선

- `transitDisruptionHandler`에 방향(`updnLine`)·종착역(`bstatnNm`) 필드 추가
- **단점**: 실시간 API 특성상 미래 시각(23:40) 검증 여전히 불가
