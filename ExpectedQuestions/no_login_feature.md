# Q. 왜 로그인 기능을 구현하지 않았나? 즐겨찾기 기능도 넣을 수 있었을 텐데.

## 핵심 답변

3가지 이유로 로그인 기능을 의도적으로 제외했습니다.

---

## 1. 개발 일정

중간 발표 전까지는 구현이 아닌 MCP 개념 스터디에 집중했습니다.
실제 개발은 중간고사 종료 후 약 1주일 뒤에 시작했으며,
남은 기간 안에 핵심 기능(Agent 동적 툴 선택)의 완성도를 높이는 것이
즐겨찾기보다 우선순위가 높다고 판단했습니다.

## 2. 인증 인프라 역량 부족

로그인 구현 시 AWS Cognito 도입을 검토했습니다.
팀 내 AWS 실무 경험자가 없었고, 팀원 중 이번 학기 클라우드 컴퓨팅을
수강 중이었지만 최종 발표 기준 3주 전에 이론 학습을 마친 상태였습니다.
AWS 서비스는 이론 이해만으로는 부족하고, 콘솔 UI와 설정 흐름에
몸으로 익숙해지는 시간이 별도로 필요합니다.
이 학습 비용까지 일정에 포함하면 리스크가 크다고 판단해 제외했습니다.

## 3. 진입장벽 최소화 (UX 관점)

로그인 없이 바로 사용할 수 있는 구조는 의도된 UX 선택이기도 합니다.
회원가입·로그인 과정은 첫 사용자에게 마찰(friction)을 유발합니다.
막차 위험 탐지는 "지금 당장 판단이 필요한" 실시간 서비스이므로,
앱을 열자마자 바로 질문할 수 있는 구조가 더 적합합니다.
즐겨찾기 같은 편의 기능은 서비스가 안정화된 이후 추가해도 늦지 않습니다.

---

## 보완 가능한 대안 (로그인 없이 검색 기록 저장)

로그인 없이도 검색 기록을 저장하는 방법은 존재합니다.
브라우저의 **localStorage**를 활용하면 서버나 계정 없이 기기 내에 데이터를 저장할 수 있습니다.

> **쿠키 vs localStorage 차이**
> - **쿠키(Cookie)**: 주로 인증 토큰 등 소용량 데이터 저장용 (최대 4KB), 서버로 자동 전송됨
> - **localStorage**: 브라우저 내 영구 저장소 (최대 5~10MB), 서버 전송 없음, 검색 기록처럼 클라이언트 전용 데이터에 적합

즉, 검색 기록 저장에는 쿠키보다 **localStorage가 더 적합**합니다.

```javascript
// 검색 기록 저장 예시
const history = JSON.parse(localStorage.getItem("searchHistory") || "[]");
history.unshift({ query: userMessage, timestamp: Date.now() });
localStorage.setItem("searchHistory", JSON.stringify(history.slice(0, 20)));
```

이 방식은 로그인 없이도 구현 가능하며, 만약 시간이 허락했다면 도입을 검토했을 것입니다.
